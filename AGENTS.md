# AGENTS.md

This file provides guidance to Codex (Codex.ai/code) when working with code in this repository.

## What this is

A macOS desktop app (Electron + React + TypeScript) for managing 4K video footage ("素材") from action cameras (DJI Pocket3/Action6/Go3S, iPhone). Footage is organized into one-level **folders** (a folder holds video files directly), browsed/filtered/tagged, then assembled into **programs** ("节目") for editing. Source clips are referenced in place — never moved; only the program **copy** action duplicates files. UI language is Chinese; macOS aesthetic.

## Commands

```bash
npm run dev      # electron-vite dev — HMR for renderer; main/preload rebuild on change
npm run build    # type-check + bundle to out/ (no packaging)
npm run dist      # build + electron-builder → dist/视频素材管理-<ver>-arm64.dmg
npm run rebuild  # rebuild better-sqlite3 against Electron's ABI (also runs on postinstall)
```

There is **no test suite and no linter** configured. "Verifying" a change means running `npm run dev` and exercising the UI.

**Gotcha:** `electron-vite dev` reliably HMRs the renderer but does not always restart the **main** process after edits to `src/main/**`. After changing main-process code (especially `db.ts` / `media.ts`), kill and relaunch the dev server, otherwise migrations and IPC changes won't take effect.

## Architecture

Three TypeScript build targets (`electron.vite.config.ts`, split tsconfigs: `tsconfig.node.json` = main+preload+shared, `tsconfig.web.json` = renderer+shared):

- **`src/main/`** — Node/Electron main process. Owns the SQLite DB, ffmpeg/ffprobe, the filesystem, and all business logic.
- **`src/preload/`** — `contextBridge` exposes a single typed `window.api` object (`Api` type) wrapping `ipcRenderer.invoke` for every channel plus three event subscriptions (`onImportProgress`, `onMaterialsChanged`, `onCopyProgress`, each returns an unsubscribe fn).
- **`src/renderer/`** — React 18 + Tailwind + Zustand. No router; one window.
- **`src/shared/types.ts`** — the data model shared by all three targets. **Start here** to understand the domain.

Data flow is strictly: renderer → `window.api.*` → `ipcMain.handle` (`src/main/ipc.ts`) → `db.ts`/`importer.ts`/`media.ts`. The renderer never touches the filesystem or DB directly.

### State & persistence

- **DB:** `better-sqlite3` (synchronous), WAL mode, schema + all queries in `src/main/db.ts`. Tables: `folders`, `materials`, `tags`/`material_tags`, `stages`, `programs`/`program_materials` (with `sort_order`), `devices`, `settings` (key/value, also used for `lastView` and migration flags).
- **User data lives outside the repo** in `app.getPath('userData')` (`~/Library/Application Support/视频素材管理/`): `library.db` + `thumbnails/<materialId>.jpg` (see `src/main/paths.ts`). Deleting the repo does not delete a user's library.
- **Renderer state:** single Zustand store `src/renderer/src/store.ts`. `bootstrap()` restores the last viewed folder/program from the `lastView` setting. `buildFilter()` translates the UI `Filters` into a `MaterialFilter` for the main process.

### Import pipeline (`src/main/importer.ts`)

Designed so adding a folder of hundreds of clips never blocks the UI. Three staged phases, each emitting `import:progress` + `materials:changed` events:

1. **scan** — `importFolder()` lists video files (`VIDEO_EXT`), inserts DB *stubs* immediately, returns `folderId`, then kicks off `processFolder()` async.
2. **meta** — `probeMeta()` (ffprobe) fills width/height/codec/duration/bitrate + shoot time; fast, batched progress every 5 items.
3. **thumbs** — `makeThumbnail()` (ffmpeg, one frame) one at a time; slow, batched every 3.

`resumePending()` runs on every launch and re-processes any material left with `meta_ready=0` or `thumb_ready=0`, so an interrupted import self-heals. While a folder is importing, the store **ignores active filters** so newly-scanned clips appear live.

### media:// protocol

Local thumbnails and 4K video are served to the renderer via a custom **privileged** scheme registered in `src/main/index.ts` (`registerSchemesAsPrivileged` + `protocol.handle`), proxying to `net.fetch(pathToFileURL(...))` — this gives range-request streaming for video. Renderer builds URLs with `mediaUrl(path)` (`src/renderer/src/format.ts`) = `media://file/<encodeURIComponent(absPath)>`. Never reference local files with `file://` in the renderer.

> **Known issue:** Chromium may fail to play H.265/HEVC video ("Unsupported pixel format: -1"). Thumbnails (generated by the bundled ffmpeg CLI, independent of Chromium) are unaffected.

### Domain rules to preserve

- **Material status** is a 3-state machine: `unused` → `used` → `deleted` (`MaterialStatus`). `used` is derived: `recomputeUsage()` sets `used` when a material is in ≥1 program, else `unused` — but **never** overrides `deleted`. Always call `recomputeUsage()` after changing program membership or restoring from `deleted`.
- **Shoot time** (`src/main/filename.ts`): parsed from the filename first (`DJI_YYYYMMDDHHMMSS_...`, 14-digit or 8-digit fallback), then file mtime. `period` (dawn/am/pm/night) is bucketed by hour. Folders auto-inherit `shoot_date` from the first parsed clip.
- **Sorting:** folder view always sorts by shoot time; **program** view supports `manual` (drag-reordered `program_materials.sort_order`) vs `time`. The program **copy** uses manual order, falling back to time order, renaming files `M001_<original>` by sequence.

### Packaging (`electron-builder.yml`)

Native binaries **must** be in `asarUnpack` (`better-sqlite3`, `ffmpeg-static`, `ffprobe-static`). `src/main/media.ts` rewrites the binary path `app.asar` → `app.asar.unpacked` at runtime (`fixPath`) — if you add another bundled native binary, add it to `asarUnpack` and apply the same rewrite. The DMG is unsigned/un-notarized (`identity: null`), arm64 only.
